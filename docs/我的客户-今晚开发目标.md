# 「我的客户」今晚开发目标（WABA 凭证暂缺版）

> 背景：WABA 认证尚未下来。今晚在**无凭证**状态下把功能写到可演示、可验收；明早只补凭证和 Meta 后台配置，不改代码。
> 交付判定：今晚结束时，通过模拟器注入的假进线能完整走通「进线 → 建档 → 时间线 → 意向分 → 回复(dry-run) → 熔断」全链路。

## 口径确认（针对实现疑问的定稿答复，以此为准）

1. **存储：今晚直接落 PocketBase**。业务代码只 import `server/storage/index.ts` 的 `store`（DataStore 接口），不直接碰 PB SDK——这是项目现有约定（见 `datastore.ts` 头注释）。不要另起本地 JSON。
2. **租户字段统一 `tenantId`**（string，非 relation），对齐 `Identity.tenantId` 和现有全部 collection（如 `social_accounts`）。本文档下方表格已按此修正。
3. **API 前缀统一 `/api/overseas/customers`**，dev 模拟器挂 `/api/overseas/dev/wa/*`。webhook 真实地址是 `/api/overseas/channels/webhook/whatsapp/:id`（channelsRouter 挂载点）。
4. **channels 配置继续留在 `data/channels.json`，今晚不迁**。`customers.channelId` 存字符串 ID 即可，不建 relation——channels 是配置数据不是业务数据，混合方案可接受。
5. **`wa_id` 租户内唯一：建复合唯一索引**。PB 迁移里 collection 的 `indexes` 数组支持 `CREATE UNIQUE INDEX idx_customers_tenant_wa ON customers (tenantId, wa_id)`。业务层 upsert 仍然先查后写，并捕获唯一冲突时重查（webhook 并发防御）。
6. **LLM 封装：`server/agents/llm.ts` 存在**，导出 `callLLM(prompt, { backend?, model?, systemPrompt? })`，支持 gemini/qwen。直接复用，禁止新建封装。
7. **24h 窗口按 `last_inbound_at`（用户最后一条 inbound 消息）计算**。这就是 Meta 客服窗口的官方定义，不是简化；conversation 计费分类是另一回事，与能否自由回复无关。
8. **pending 消息对用户可见**：时间线正常展示 + 「待发送」角标（生产环境凭证失效时同样有意义，不是 dev-only）。明天凭证补齐后走 `resend-pending` 一键重发。
9. **`stage` 与 `sop_step` 拆成两个字段**。`stage` 保留现有 `潜客/询盘中/已报价/成交/沉默30/沉默60`（前端四视图依赖它，是"客户处于什么状态"，可回退）；`sop_step` 新增 `首响/问需求/报价/样品打样/付款/履约/复购唤醒`（是"流程推进到哪一步"）。今晚 `stage` 必做，`sop_step` 建字段、UI 明天做。
10. **意向阈值默认值**：`0–50 auto（自动回目录）`、`51–84 confirm（草稿待确认）`、`85+ manual（人工接管审查）`。**大单默认**：预估金额 ≥ $1,000 或明确数量且单价推算 ≥ $1,000。**今晚熔断条件只做三个**：`call_request`、大单、投诉；砍价僵持/指名老板/付款PI请求等进词库配置留扩展。所有阈值写成常量并从租户配置可覆盖（见文末"配置分层"）。
11. **模拟器鉴权**：`NODE_ENV !== 'production'` 不挂载 + 路由内再校验现有 auth middleware，双保险。
12. **今晚 UI 改动范围严格三点**：回复输入框、24h 窗口倒计时、待发送角标。布局不动，不做 SOP 台。「拉起 WhatsApp 通话」改为 `wa.me` 深链，文案改「打开 WhatsApp 联系」（deep link 只能打开聊天，不能直接拨语音）。
13. **订单关联今晚不做**：保留 `orderHistory` 展示字段（json），M4 再接「我的订单」。
14. **referral 缺失的兜底**：`first_source` 允许为空；详情页提供手动选择来源（下拉：WhatsApp直连/Instagram/TikTok/Facebook/Alibaba/其他）；wa.me 链接可在 `text` 预填参数里带来源码（如 `ref:IG-bio`），解析器识别后写入。

## 零、总体架构约定（先读这个）

1. **凭证只存在于 HTTP 边界**。`server/integrations/whatsapp.ts` 的发送函数和 webhook 验证是唯一碰凭证的地方。中间所有业务逻辑（落库、评分、熔断、前端）只依赖消息数据，禁止直接 import axios 调 Graph API。
2. **凭证为空 = dry-run 模式**。发送层检测 channel config 里 `accessToken` 为空时：不发 HTTP 请求，消息照常写 `wa_messages`（direction=out，status=`pending_credentials`）和 `timeline_events`。明天填凭证后，新消息自动真实发送；存量 pending 消息提供一个"重发"接口。
3. **webhook 解析器严格按 Meta Cloud API 官方 payload 结构写**（`entry[].changes[].value` 下的 `contacts` / `messages` / `statuses`，消息内含 `referral` / `context`）。不要自造格式——模拟器和真实进线必须走同一个解析函数。
4. **所有新表带 `tenantId` 字段**（对齐 `Identity.tenantId` 与现有 collection 命名）。
5. 意向分必须带 `signals[]` 加分明细，保持可解释（前端 UI 已按此设计）。

## 八、对外销售 Prompt（T7 配套，AI 托管自动回复用）

> 现有 `agentChat.ts` 的 SYSTEM_PROMPTS 是**对内助手** prompt（输出 Markdown 给卖家看），不能用于回复买家。新建 `server/prompts/salesPrompts.ts`（与现有 prompts 目录并列）：

- **角色**：你是 {公司名} 的外贸销售代表（不是"AI助手"），用买家的语言回复，口语化、单条 ≤300 字符、无 Markdown、无套话。
- **上下文注入**：复用 `buildEnterpriseContext()`（企业知识库）+ customer_insights（语言/产品/阶段/missing_fields）+ 最近 10 条对话 + referral 广告素材（首响时引用客户看到的 headline）。
- **五个场景 skill**：首响问候 / 目录推品 / 追问缺失字段（一次只问1–2个）/ 报价（守 minMargin）/ 老客唤醒（结合节日与历史偏好）。按 sop_step 和触发原因选择。
- **护栏（必须实现）**：
  1. 只引用知识库事实，没有的信息回"确认后回复"并转草稿，禁止编造交期/认证/定制能力；
  2. 无价格数据不报价，降级草稿；
  3. 生成前熔断自检（通话请求/投诉/法律风险 → 不回复，直接升级）；
  4. 遵守 `brand.taboos` 禁忌；
  5. LLM 输出结构化 JSON `{ reply_text, confidence, should_escalate, reason }`，confidence < 阈值自动降级为草稿待确认，不直接发送。
- 今晚至少实现：首响 + 目录推品两个 skill，走 dry-run 链路可在时间线看到 AI 生成的回复文本。

### Skill 的来源与迭代（今晚执行口径）

1. **今晚 skill 初稿的写法**：用经典 B2B 销售方法论搭骨架（SPIN 式追问：一次1–2个问题；报价留砍价空间；先确认需求再报价），结合 enterprise.json 里的品类/市场/价格策略做参数化。不追求一步到位准确，追求结构可迭代。
2. **今晚必埋的数据飞轮**：`wa_messages` 出站记录加一个 `ai_draft` 字段——AI 原始草稿和实际发送文本分开存。业务员每次改草稿再发送，diff 就是免费标注。这是后续 skill 变准（乃至将来微调）的唯一数据来源，今晚不埋后面补不回来。
3. **今晚顺手做的评测集**：`server/prompts/salesEval.ts` 放 20 条典型询盘（只问价/要目录/明确大单/要通话/投诉/阿语/西语/砍价/催样品/沉默唤醒回复），每条标注期望行为（该回什么类型、该不该升级）。改 prompt 后跑一遍防回归，用模拟器注入即可。
4. **后续（不是今晚）**：导入老业务员真实聊天记录蒸馏话术模式；试用 SleekFlow/Wati/respond.io 抄通用 playbook；`agentLearning.userCorrections` 定期把积累的修改蒸馏回 prompt。**不研发/不微调大模型**——调用量和数据量到位之前，前沿模型 + prompt + 知识库注入是行业标准做法。

### 九、Playbook 采集设计（skill 原料的产品化口径）

四种原料对应四种采集形态，注意：**只有"事实"用表单，"经验"和"行为"都不用表单**。

| 原料 | 形态 | 设计要点 |

**校准不走表单**，三个零负担行为信号：①草稿修改 diff（`ai_draft` 字段今晚已埋）②成交/流失结果回标（成交客户对话链路上的话术自动加权）③老板审校队列（对 AI 已发消息点好/差）。评测集（salesEval.ts）是开发侧资产，商家不可见。

## 配置分层：哪些是企业个体信息（表单录入），哪些是行业通用（大盘默认值）

产品逻辑类问题（意向阈值、大单定义、熔断条件、缺失字段等）不是"问出一个全局正确答案"，而是要拆成两层：

### A. 企业个体信息 —— 需要表单录入，不能用默认值

> 落点：**扩展现有企业信息体系**（`data/enterprise.json` + `EnterprisePage.tsx` + `server/routes/enterprise.ts` 已有完整的表单-存储-读取链路），新增 `customerOps` 一节，不要另建配置系统。今晚 T7 读这里，读不到就用 B 层默认值。

| 配置项 | 为什么必须个体录入 | 现有字段基础 |
|---|---|---|
| 大单金额/数量阈值 | 500 pcs 对假发是 $2,000+ 大单，对小商品可能不到 $100，跨类目无法通用 | `products.priceRange`、`products.moq` 已有，可推算默认值再让用户改 |
| 产品目录、MOQ、样品政策 | 自动回目录和报价草稿的原材料，完全企业私有 | `products.*` 已有 |
| 报价要素（价格梯度、付款方式、交期） | 报价草稿生成必需 | `operations.leadTime`、`operations.paymentTerms` 已有 |
| 熔断后接管人及通知渠道 | 老板是谁、推钉钉还是飞书，每家不同 | 需新增（channels 配置里已有钉钉/飞书 webhook 可选） |
| 自动化档位偏好 | 有的老板要求全部人工确认，有的接受低分全自动 | `strategy.agentAutonomy` 已有雏形 |
| 高/低质信号补充 | 每家对"好客户"的定义不同（如只做整柜、不接样品单） | `customers.highValueSignals` / `lowQualitySignals` 已有 |
| 产品相关缺失字段 | 是否需要 Logo 定制、包装选项，取决于品类 | `operations.customization` 已有 |

### B. 行业通用 —— 直接内置默认值（可被 A 层覆盖）

> 落点：写成 `server/lib/waDefaults.ts` 常量表，注释里标注依据。读配置时 `tenant 配置 ?? 行业默认`。
